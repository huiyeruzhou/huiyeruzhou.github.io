Codex-to-IM 当前架构 ​

本文描述 Codex-to-IM 当前真实运行架构、核心概念、数据归属和主要工作流。它不是未来规划文档，而是用来回答两个问题：

  • 现在系统如何把本地 Codex 线程接到 IM。
  • codex_thread_id -> BridgeSession -> ChannelChat -> IMChannel 这条链路里，每一层到底负责什么。

当前产品形态是本地可安装应用：

  • codex-to-im open：启动本地 Web 工作台，并尝试启动 bridge。
  • codex-to-im start：只启动 bridge。
  • codex-to-im status、url、stop、autostart、uninstall：本地服务管理命令，由 src/entrypoints/cli.ts 和 src/local-service/manager.ts 实现。
  • 配置和运行数据放在 ~/.codex-to-im；项目不要求安装流程写入 ~/.codex/config.toml。

核心身份链路 ​

Codex-to-IM 的核心不是“一个 IM 聊天等于一个 Codex 线程”。正确链路是：

text
Codex thread
  -> BridgeSession
  -> ChannelChat
  -> IMChannel / IM chat

更具体地说：

text
~/.codex/sessions/**/*.jsonl
        |
        | thread id
        v
BridgeSession.codex_thread_id
        |
        | BridgeSession.id
        v
ChannelChat.bridgeSessionId
        |
        | channelType + chatId
        v
飞书里的某个聊天

这条链路有几个关键规则：

  • codex_thread_id 只属于 BridgeSession。
  • ChannelChat 不保存 Codex thread id，只保存 bridgeSessionId。
  • IM 消息进来时，bridge 先按 channel 和 chat 找到 ChannelChat，再读取 ChannelChat 指向的 BridgeSession，最后从 session 上取 codex_thread_id 来继续或镜像 Codex 线程。
  • 一个 BridgeSession 可以先存在，之后才产生 codex_thread_id。
  • 一个 IM chat 只保存一个 ChannelChat；一个 BridgeSession 也只保留一个聊天绑定。
  • 一个通道实例只是消息入口，不等于会话，也不等于 Codex thread。

Codex Thread ​

Codex thread 是 Codex 自己的对话线程。Codex-to-IM 不拥有它，只记录它的 id。

Codex thread 可以来自不同入口：

  • IM 里发起的一次 SDK turn。
  • 本地 Codex CLI / TUI 会话。
  • 本地 Codex Native 客户端产生的会话。
  • 被 /t 从本地 Codex session 列表接管的已有线程。

Codex-to-IM 读取 Codex 自己的数据：

  • ~/.codex/sessions/**/*.jsonl：Codex 会话事件。
  • ~/.codex/state_*.sqlite：本地 session 列表兼容元数据，只读使用。

这些文件不属于 Codex-to-IM。Codex-to-IM 可以读取、索引、展示和镜像它们，但不应把它们当作自己的主存储。

BridgeSession ​

BridgeSession 是 Codex-to-IM 自己拥有的本地产品会话，保存在：

text
~/.codex-to-im/data/sessions.json

它用本地 BridgeSession.id 作为主键，负责承载“这个可持续对话在 Codex-to-IM 里的状态”。

它保存：

  • 会话名。
  • 工作目录。
  • 模型、模式、reasoning、sandbox、network、provider 等运行设置。
  • 可选的 codex_thread_id。
  • 运行状态、队列状态、健康状态。
  • 流式 UI 状态、mirror 状态和诊断信息。
  • Bridge 自己的消息缓存索引。

BridgeSession 和 Codex thread 的关系是“本地会话容器指向 Codex thread”：

  • 没有 codex_thread_id：这个 Bridge 会话还没有真实 Codex thread，可能只是一个新建会话、草稿会话或尚未发起过 turn 的会话。
  • 有 codex_thread_id：这个 Bridge 会话已经关联一个真实 Codex thread，后续 history、mirror、resume 都通过这个 id 找 Codex 数据。

旧字段不再参与业务判断：

  • sdk_session_id
  • sdkSessionId
  • desktop_thread_id
  • desktopThreadId
  • thread_origin
  • threadOrigin
  • thread_id
  • threadId

这些字段只允许作为迁移输入出现；运行时代码不应把它们当成 fallback。

ChannelChat ​

ChannelChat 是“某个 IM chat 绑定到哪个 BridgeSession”的关系，保存在：

text
~/.codex-to-im/data/channel-chats.json

它保存：

  • 通道身份：channelType、可选 provider、通道别名。
  • 聊天身份：chatId、可选聊天类型 chatKind、可选用户 id。
  • bridgeSessionId：指向一个 BridgeSession.id。

ChannelChat 不拥有 Codex thread 身份，也不保存工作目录、模型、模式、显示名等运行时字段。它的职责只到 bridgeSessionId 为止。

普通消息的解析路径是：

  1. 飞书 adapter 收到一条消息。
  2. bridge 根据 channel instance 和 chat id 找到当前聊天的 ChannelChat。
  3. ChannelChat 给出 bridgeSessionId。
  4. store 读取对应 BridgeSession。
  5. 如果 session 有 codex_thread_id，就 resume 对应 Codex thread。
  6. 如果 session 没有 codex_thread_id，就创建或启动新的 Codex turn，并把返回的 thread id 写回 BridgeSession.codex_thread_id。

因此，ChannelChat 是 IM 世界和 Bridge 会话世界之间的连接点，不是 Codex thread 的存放点。

IMChannel ​

IMChannel 是配置好的 IM 入口。当前支持飞书。

一个通道实例包含：

  • provider：飞书。
  • 实例 id。
  • 别名。
  • 是否启用。
  • 平台凭据和连接配置。
  • 运行期 adapter 状态。

多个通道实例可以同时存在，例如：

  • 飞书主号。
  • 飞书备份号。

通道实例不是会话。它只是让某个平台的消息进入 bridge，并把 bridge 输出送回平台。真正的会话身份仍然在 BridgeSession，聊天到会话的关系仍然在 ChannelChat。

一条 IM 消息如何进入 Runtime ​

普通 IM 文本消息进入当前 runtime 的主路径：

  1. 平台 adapter 接收飞书消息，生成 InboundMessage。
  2. bridge-manager.ts 判断它是权限回调、命令还是普通 prompt。
  3. 如果是普通 prompt，channel-router.ts 解析当前 chat 的 ChannelChat。
  4. ChannelChat 指向 BridgeSession.id。
  5. interactive-turn/runner.ts 建立本次 turn 的任务状态和流式反馈。
  6. CodexRoutingProvider 根据会话 runtime 与 provider 设置路由：Codex 支持 sdk、pty、tmux，Claude 支持 pty、sdk。
  7. provider 创建或继续对应 runtime 的会话身份，例如 Codex thread 或 Claude session_id。
  8. 如果 provider 返回 runtime identity，写入 BridgeSession.runtime 对应容器。
  9. delivery pipeline 把最终文本、卡片、附件等发回对应 IM channel。

这条路径保证“IM 只是入口，BridgeSession 才是本地会话，Codex thread 才是真正的 Codex 对话”。

/t 接管本地 Codex 线程 ​

/t 命令用于把本机可发现的 Codex thread 接入 IM。

典型过程：

  1. 用户发送 /t。
  2. local Codex session index 扫描 ~/.codex/sessions/**/*.jsonl。
  3. bridge 返回最近的本地 Codex 线程列表。
  4. 用户发送 /t 1。
  5. 被选中的 Codex thread 会被 materialize 成一个 BridgeSession，或复用已有同 thread 的 BridgeSession。
  6. 当前 IM chat 创建或更新 ChannelChat，让它指向这个 BridgeSession.id。
  7. 后续普通消息通过 ChannelChat 找到 session，再通过 BridgeSession.codex_thread_id 继续同一条 Codex thread。

因此 /t 不是“把 thread id 写到聊天上”，而是“把 Codex thread 纳入 BridgeSession，再让聊天绑定这个 BridgeSession”。

Mirror 运行时 ​

Mirror 是对本地 runtime JSONL 文件的持续观察。Codex 和 Claude Code 各自有底层 JSONL source，但共用 mirror-runtime.ts、mirror-turns.ts 和 mirror-feedback-controller.ts 这套 reducer / delivery 管线。

主路径：

  1. mirror-subscription-registry.ts 找到有本地 runtime 身份的 BridgeSession。
  2. 再找到绑定这些 session 的 ChannelChat。
  3. mirror-runtime.ts 根据绑定关系建立或更新订阅。
  4. 每个订阅通过当前 runtime 的 source 定位对应 JSONL 文件。
  5. 文件监听和定时 reconcile 读取新增记录。
  6. suppression 去掉当前 IM 发起 turn 造成的重复回显。
  7. mirror-turns.ts 把 JSONL 事件合并成可展示 turn。
  8. mirror-feedback-controller.ts 更新流式卡片或发送最终 fallback 消息。
  9. 如果这批记录包含可被当前 active turn 接受的 task_complete / task_aborted，routeRuntimeRecords 会通过 TurnCoordinator.claimRuntimeTerminal 把终态归还给 active turn。Codex 和 Claude Code 使用同一个接口；普通 Claude IM turn 仍保留 mirror 卡片作为首屏 反馈路径，不会被 terminal claim 抢成 im: 流。

Codex mirror 的身份链路是：

text
codex_thread_id -> BridgeSession -> ChannelChat -> channel/chat

Claude Code mirror 的身份链路是：

text
runtime.claude.sessionId + runtime.claude.cwd -> BridgeSession -> ChannelChat -> channel/chat

如果 JSONL source 连续多次无法定位，mirror-runtime.ts 会通过 runtime-neutral identity cleanup hook 清理当前 runtime 身份：Codex 清理 runtime.codex.threadId，Claude Code 清理 runtime.claude.sessionId / runtime.claude.cwd。这样 dangling 本地线程不会在 subscription registry 里反复重建。

解析出来。routeCodexRecords 和 claimCodexTerminal 仍作为兼容入口保留， 新代码应优先使用 runtime-neutral 的 routeRuntimeRecords 和 claimRuntimeTerminal。

命令应用层 ​

Slash command 主要由 src/bridge/command/dispatch.ts 分发。当前 command 目录已经按一部分工作流拆分：

  • session-thread.ts：线程选择、切换、重命名；纯参数解析、命名校验和 /clear 确认卡片在 session-args.ts。
  • runtime-settings.ts：mode、reasoning、sandbox、network、model、工作目录和 UI detail 等运行设置入口；同时 re-export 旧 provider/bootstrap helper 入口。
  • provider-settings.ts：/provider 的 Codex/Claude provider 切换编排。
  • runtime-bootstrap.ts：Codex thread 预创建和 SDK SSE thread id 解析。
  • runtime-session.ts：runtime session 映射、创建、格式化和 provider 阻断提示 helper。
  • tmux.ts：tmux provider 控制；纯 tmux 参数、按键和 screen 参数解析在 tmux-args.ts。
  • pty.ts：pty provider 屏幕观察和定时刷新。
  • shell.ts：远程 shell 命令执行。
  • auto.ts：自动任务。
  • status.ts、diagnostics.ts：状态和诊断。
  • global-settings.ts：全局设置。
  • hot-update.ts：热更新。
  • presentation/：命令展示格式；根部 presentation.ts、auto-presentation.ts、diagnostics-presentation.ts 仅保留兼容 re-export。

命令可以分为三类：

  • 通道级命令：不要求当前 chat 已经有 session，例如 /status、/t。
  • 会话级命令：读取或修改当前 chat 的 ChannelChat/session，例如 /new、/t 1、/provider、/his、/stop。
  • 全局配置命令：通过某个 chat 触发，但实际修改全局设置，例如 /ui、/his limit。

Command 重构的当前完成标准是模块边界优先，而不是继续把每条命令拆得很细：

  • bridge/host 是生产代码里唯一直接调用 bridge/command 的上层 owner。
  • channels、bridge/turn、runtime、storage、operator-ui 不直接依赖 bridge/command。
  • 跨命令和 turn 共用的回调协议放在 src/bridge/callbacks/，避免 turn 反向依赖 command。
  • command 不直接 import bridge/host/*：全局 bridge context 位于 src/bridge/context.ts，会话绑定 router 位于 src/bridge/session/channel-router.ts，startup notice target 位于 src/bridge/startup-notice-target.ts；src/bridge/host/* 同名文件仅保留兼容 facade。

本地工作台 ​

本地 Web 工作台由 src/operator-ui/server.ts 提供页面和 API。

主要页面：

  • 概览：显示 UI、bridge、通道数量和进程状态。
  • Sessions：列出 Bridge 会话和本机可发现的 Codex 线程。
  • Session History：查看某个 session 的历史。
  • Channels：管理飞书通道实例、ChannelChat、默认目标和测试。
  • Config：编辑全局默认设置。
  • 命令：展示 IM 命令帮助。

UI 里的可变操作应使用 BridgeSession.id。本机 Codex thread 在未 materialize 前只能作为只读 local-index row 展示；一旦用户对它执行重命名、绑定、配置、删除或设置默认目标等操作，就应先 materialize 成 BridgeSession。

数据归属 ​

Codex-to-IM 自有数据位于 ~/.codex-to-im：

  • config.v2.json：结构化配置。
  • config.env：兼容旧工具的配置快照和 bootstrap 输入，不是多实例通道配置的完整来源。
  • data/sessions.json：BridgeSession，包含唯一的 codex_thread_id。
  • data/channel-chats.json：ChannelChat，只保存 IM chat 身份和 bridgeSessionId。
  • data/channel-default-targets.json：通道实例的默认目标。
  • data/messages/<sessionId>.json：Bridge 消息缓存。
  • data/permissions.json：权限回调链接。
  • data/offsets.json：adapter 消费偏移。
  • data/dedup.json：去重时间戳。
  • data/audit.json：审计记录。
  • thread-table-messages.json：线程卡片消息记录。
  • bridge 和 UI 的 runtime 状态文件。

Codex 自有数据仍位于 ~/.codex。Codex-to-IM 只读使用它们。

迁移边界 ​

启动迁移负责把旧 session thread 身份字段收敛到 BridgeSession.codex_thread_id。旧 data/bindings.json 到 data/channel-chats.json 的迁移使用显式脚本执行，不在运行时代码中保留兼容读取：

  • session 上旧的 sdk_session_id、desktop_thread_id、thread_id 会迁移成 codex_thread_id。
  • scripts/migrate-bindings-to-channel-chats.js 只保留旧 binding 中 active: true 的记录，并把旧 binding 上的运行时字段迁到它指向的 session 空字段；成功写入 channel-chats.json 后会删除旧 data/bindings.json。
  • 旧 selector 字符串只允许作为迁移输入；运行时代码应使用 BridgeSession.id 和显式的 Codex thread import。

迁移后的不变量：

  • Codex thread 身份只在 session。
  • ChannelChat 只指向 session。
  • channel 只负责平台入口和消息投递。

术语规则 ​

  • 使用 codex_thread_id 表示唯一持久化的 Codex thread 身份。
  • 不新增 thread_id 作为替代字段。
  • 不把 sdk_session_id、desktop_thread_id、thread_origin 当运行时 fallback。
  • 用户文案可以使用“本地 Codex 会话”或“Codex 线程”。
  • 内部业务模型围绕 BridgeSession、ChannelChat、IMChannel 和 codex_thread_id 表达。
  • ChannelChat 是 chat 到 BridgeSession 的关系，不是 chat 到 Codex thread 的关系。
