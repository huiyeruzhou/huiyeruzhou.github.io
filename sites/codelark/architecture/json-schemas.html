JSON Schema 与升级契约 ​

Codex-to-IM 为 ~/.codex-to-im 下的本地数据文件发布 JSON Schema。入口文件是 schemas/manifest.json。

这些 schema 主要服务三件事：

  • 在 bridge 读取前校验用户手动编辑过的本地文件。
  • 给 doctor 或未来的 migrate 命令提供机器可读的文件清单。
  • 明确持久化数据版本之间的升级边界。

运行时，Codex-to-IM 会在加载 bridge store 前执行非 ChannelChat 的启动迁移。data/bindings.json 到 data/channel-chats.json 的迁移是一次性显式迁移，使用 scripts/migrate-bindings-to-channel-chats.js 执行，运行时代码不再读取旧 binding 文件。

文件清单 ​

运行时文件Schema说明
config.v2.jsonschemas/config.v2.schema.json结构化配置，版本字段是 schemaVersion: 2。
data/sessions.jsonschemas/data/sessions.v1.schema.json以 Bridge session id 为 key 的 map；codex_thread_id 保存在这里。
data/channel-chats.jsonschemas/data/channel-chats.v1.schema.json以 ChannelChat id 为 key 的 map；使用 bridgeSessionId 指向 session，禁止保存 thread 身份和旧 binding 运行时字段。
data/channel-default-targets.jsonschemas/data/channel-default-targets.v2.schema.json以 channel instance id 为 key 的默认目标 map；使用 bridgeSessionId，启动时会丢弃缺少它的旧记录。

升级模型 ​

JSON Schema 本身不执行迁移。schemas/manifest.json 是迁移器应遵守的契约：

  1. 读取 schemas/manifest.json。
  2. 将每个 files[].path 解析到 ~/.codex-to-im 下。
  3. 如果文件不存在，按 missingFile 策略处理。
  4. 如果文件有 versionField，从该字段识别当前版本；否则按 manifest 条目声明的版本处理。
  5. 用源版本 schema 校验原始文档。
  6. 按版本顺序执行注册迁移。
  7. 用当前版本 schema 校验迁移后的文档。
  8. 通过临时文件加 rename 原子写回。

迁移代码应尽量保留未知字段，除非 manifest 或 schema 明确标记这些字段已废弃。这样可以降低降级、热更新和手工搬运本地数据时的数据丢失风险。

Thread 身份迁移 ​

当前启动迁移主要修复 thread 身份重构。

迁移前，旧数据里可能出现多种 thread 字段：

  • sdk_session_id
  • sdkSessionId
  • desktop_thread_id
  • desktopThreadId
  • thread_origin
  • threadOrigin
  • thread_id
  • threadId
  • 旧 binding 上的 codex_thread_id
  • 旧 binding 上的 codexThreadId

迁移后的目标只有一个：

text
BridgeSession.codex_thread_id

具体规则：

  • session 上的 sdk_session_id、desktop_thread_id、thread_id 会在 session 没有 codex_thread_id 时折叠进去。
  • 旧 binding 到 ChannelChat 的迁移不在启动路径执行；显式脚本只保留 active: true 的旧记录，并把 workingDirectory、model、mode、chatDisplayName 迁到对应 session 的空字段，成功后删除旧 data/bindings.json。
  • 旧 data/ui-session-meta.json 里的名称会合并到 sessions.json.name；只有本地 Codex thread 名称而没有 BridgeSession 时不会再创建新 BridgeSession，然后删除 ui-session-meta.json。
  • v1 data/channel-default-targets.json 会从旧 target selector 字符串改写成 bridgeSessionId。如果 selector 指向本地 Codex thread，会先 materialize 成 BridgeSession，再删除旧 selector 字段。

身份规则 ​

BridgeSession.codex_thread_id 是唯一持久化的 Codex thread 身份字段，必须保存在：

text
data/sessions.json

data/channel-chats.json 只能通过 bridgeSessionId 指向 session。它不应保存任何 Codex thread 身份，因为 ChannelChat 的职责是“IM chat -> BridgeSession”，不是“IM chat -> Codex thread”。

Schema 会拒绝这些已删除身份字段：

  • sdk_session_id / sdkSessionId
  • desktop_thread_id / desktopThreadId
  • thread_origin / threadOrigin
  • thread_id / threadId
  • ChannelChat 上的 codex_thread_id / codexThreadId

与架构链路的关系 ​

Schema 约束的是这条链路：

text
Codex thread id
  -> data/sessions.json 中的 BridgeSession.codex_thread_id
  -> data/channel-chats.json 中的 ChannelChat.bridgeSessionId
  -> channelType + chatId 对应的 IM chat

如果 schema 允许 ChannelChat 保存 thread id，这条链路就会出现两个 thread 身份来源，history、mirror、reuse 和 /t 切换都会变得不一致。因此 schema 层必须把 thread 身份固定在 session 上。
