Codex-to-IM 安装与使用指南 ​

本文面向需要把本机 Codex 或 Claude Code 接到飞书里使用的用户。当前推荐路径是安装本地 codex-to-im 应用，运行安装向导完成机器人和默认 runtime 配置，然后在 IM 里通过命令绑定和继续本地线程。

运行模型 ​

安装后推荐先运行：

bash
codex-to-im setup

项目不接管 ~/.codex、~/.claude-code 或 ~/.claude-code-router 的模型供应商配置。Codex CLI、Claude Code 或本机登录态应按你的环境自行配置或安装；安装向导只写入 ~/.codex-to-im/config.v2.json 和兼容快照 config.env。

飞书应用配置 ​

飞书侧需要至少完成以下配置：

  1. 在飞书开放平台创建自建应用，记录 App ID 和 App Secret。
  2. 启用 Bot 能力。
  3. 添加消息发送、消息读取、卡片读写、消息更新、云文档评论写入等权限。
  4. 发布版本并由管理员审批。
  5. 启动 codex-to-im bridge。
  6. 在飞书应用的事件订阅里选择长连接，并添加 im.message.receive_v1、drive.notice.comment_add_v1、im.chat.member.bot.deleted_v1、im.chat.disbanded_v1 事件和 card.action.trigger 回调。
  7. 再次发布版本并审批。

如果 CTI_FEISHU_ALLOWED_USERS 为空，所有能给机器人发消息的人都可以使用该 bot。需要收紧访问时，填写飞书用户 open_id 列表。

云文档评论能力除事件订阅外，还需要文档评论写入权限。缺少 docs:document.comment:create 或 docs:document.comment:write_only 时，bridge 可以收到 drive.notice.comment_add_v1，但写回评论或添加回复前的 Typing 表情会收到 Feishu 99991672 权限错误。

本地工作台 ​

常用命令：

bash
codex-to-im open
codex-to-im status
codex-to-im url
codex-to-im start
codex-to-im stop

工作台主要用于：

  • 查看 UI 和 bridge 是否运行。
  • 配置飞书通道实例。
  • 测试飞书凭据和连通性。
  • 查看本地 Codex 会话和 Codex-to-IM Bridge 会话。
  • 查看会话历史。
  • 给通道设置默认目标会话。
  • 查看日志和诊断信息。

配置主文件是 ~/.codex-to-im/config.v2.json。config.env 仍可作为初始配置和运维快照使用；程序启动时会把旧格式迁移或同步到 v2 配置。

IM 中的基础使用 ​

把机器人拉进飞书会话或直接私聊机器人后，可以使用以下流程：

  1. 发送 /status 检查 bridge 和通道状态。
  2. 发送 /t 查看最近 10 条本地 Codex 线程。
  3. 发送 /t all 查看最多 200 条本地 Codex 线程。
  4. 发送 /t 1 切换当前 IM 会话到第 1 条线程。
  5. 之后直接发送普通消息，即可继续该 Codex 线程。

新建会话：

text
/new
/new my-thread
/new my-thread ./my-project
/clear
/clear my-thread ./my-project

切换和重命名：

text
/t 1
/t archive 1
/t rename new-name

查看历史：

text
/his
/his 10
/his raw 10
/his json

调整运行设置：

text
/provider sdk
/provider pty
/provider tmux
/mode normal
/mode yolo
/reasoning 5
/model
/model <模型名>

停止当前任务：

text
/stop

发送以 / 开头的普通文本时，在前面多加一个 /：

text
//status

这会把 /status 作为普通消息发给 Codex，而不是执行命令。

提供方选择 ​

在 Codex runtime 下，默认推荐 sdk：

text
/provider sdk

sdk 使用 @openai/codex-sdk 直接运行 Codex turn，适合大多数 IM 远程对话。Codex 还支持 pty 和 tmux。

需要通过 Codex TUI 运行，但不需要手动 attach 终端时，可以使用 pty：

text
/provider pty
/pty-screen
/pty-screen 120 5s
/pty-screen stop

pty 会用伪终端启动 Codex TUI 并 resume 当前 Codex thread。Codex TUI 首次询问是否信任当前目录时，bridge 会把确认请求转发到 IM，只有用户选择信任后才会继续发送本轮 prompt。

需要接管已有终端或 TUI，并保留可 attach 的 tmux session 时，可以使用 tmux：

text
/provider tmux
/tmux-switch
/tmux-attach <session>
/tmux-screen
/tmux ...你的输入...

tmux 模式依赖本机 tmux session，适合把一个已经打开的终端会话映射到 IM。

在 Claude runtime 下，/provider 切换的是 Claude provider：

text
/runtime claude
/provider pty
/provider sdk

pty 使用本机 Claude Code TUI/JSONL mirror 路径，适合 CCR-backed Claude Code 和需要观察 TUI 的场景；sdk 使用 @anthropic-ai/claude-agent-sdk 原生事件路径。

数据和日志 ​

Codex-to-IM 自有数据位于：

text
~/.codex-to-im

常见文件：

  • config.v2.json：结构化配置。
  • config.env：兼容旧工具的配置快照。
  • data/sessions.json：Bridge 会话。
  • data/channel-chats.json：IM chat 到 Bridge 会话的绑定；一个 Bridge 会话只绑定一个聊天。
  • data/messages/：Bridge 消息缓存。
  • logs/：bridge、UI 和启动器日志。
  • runtime/：运行状态、PID、端口等临时状态。

Codex 自己的会话数据仍由 Codex 管理，Codex-to-IM 只读取：

text
~/.codex/sessions/**/*.jsonl
~/.codex/state_*.sqlite

常见问题 ​

codex-to-im open 后 bridge 启动失败 ​

先查看：

bash
codex-to-im status

再打开工作台日志页，确认飞书凭据、Codex 登录态和本机 Node.js 版本。

飞书收不到消息 ​

检查：

  • 飞书应用是否已发布并审批。
  • Bot 能力是否已启用。
  • 长连接事件订阅是否保存成功。
  • im.message.receive_v1、drive.notice.comment_add_v1、im.chat.member.bot.deleted_v1、im.chat.disbanded_v1 和 card.action.trigger 是否已添加。
  • 云文档评论回复是否缺少 docs:document.comment:create 或 docs:document.comment:write_only 权限。
  • bridge 是否正在运行。

流式卡片没有逐字输出 ​

这是正常情况。当前 Codex SDK 事件通常在 assistant 消息完成时给出最终文本，飞书流式卡片主要用于展示 thinking、工具进度和最终答案更新。

限制使用者 ​

默认 CTI_FEISHU_ALLOWED_USERS= 为空，表示不做用户白名单限制。要限制用户，设置为飞书 open_id 列表：

bash
CTI_FEISHU_ALLOWED_USERS=ou_xxx,ou_yyy

修改后重启 bridge：

bash
codex-to-im stop
codex-to-im open

卸载 ​

bash
codex-to-im stop
npm unlink -g codex-to-im

如需清理本地数据，再删除：

bash
rm -rf ~/.codex-to-im
