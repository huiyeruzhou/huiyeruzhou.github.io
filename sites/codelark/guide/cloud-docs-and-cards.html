Lark CLI、云文档、表单、Doctor 与交互卡片 JSON 注入 ​

重命名这一节，叫做飞书功能，另外什么本机依赖这些东西就不要讲了

lark-cli 与云文档评论 ​

本阶段已在 Node.js 24 环境下安装并验证 @larksuite/cli，本机 lark-cli --version 为 1.0.45。@larksuite/cli 也已加入项目依赖管理：package.json 依赖为 @larksuite/cli: ^1.0.45，package-lock.json 锁定了 node_modules/@larksuite/cli 及其 lark-cli bin。lark-cli docs --help 显示文档能力包含 docs +fetch 与 docs +update，并提示 Docs v2 应显式使用 --api-version v2。

Codex-to-IM 的云文档入口仍由飞书事件 drive.notice.comment_add_v1 触发。默认模式下，用户需要在云文档评论里 @ 机器人；Bridge 会用 doc:<file_type>:<file_token> 作为聊天 ID 建立或复用评论回复会话，把评论 file token、file type、comment id、reply id、引用内容和用户问题拼成一次普通 runtime turn。默认模式建立后，同一文档后续评论可以作为这条云文档评论会话的后续消息处理；最终回复通过 Drive 评论 reply 接口写回触发本轮输入的同一个评论线程。

云文档评论里 @ 机器人后发送 /new [群名] [目录] 会启用显式 doc-as-chat 模式。Bridge 会创建飞书群聊，把这份云文档映射到该群聊，并在群聊里承载后续普通 IM 对话。新模式建立后，云文档评论不再接入 bot 对话；如果用户继续在云文档评论里发消息，Bridge 只回复提示用户去已创建的群聊聊天。

未建立默认模式或新模式的文档评论仍必须 @ 机器人或在评论富文本内容里提及机器人，避免 bridge 把无关文档评论当成聊天消息。机器人自己写出的评论会被忽略，避免回复回环。

为了让 agent 能真的读取或修改文档正文，云文档 prompt 现在会告诉模型直接调用：

bash
lark-cli docs +fetch --api-version v2 --as bot --doc <file_token>
lark-cli docs +update --api-version v2 --as bot --doc <file_token> --mode append --markdown '<内容>'
lark-cli docs +update --api-version v2 --as bot --doc <file_token> --mode replace_range --selection-by-title '<标题>' --markdown '<内容>'

如果 lark-cli 或机器人权限不可用，模型应在云文档评论里明确说明需要补 CLI 绑定或 Drive 权限，而不是假装已经读取了文档正文。

云文档评论还会在 turn 开始前给目标 reply 添加 Typing reaction，回复完成后在 sendCloudDocumentReply() 的 finally 中删除。使用的接口是：

text
POST /open-apis/drive/v2/files/{fileToken}/comments/reaction?file_type={fileType}
body: { "action": "add|delete", "reply_id": "...", "reaction_type": "Typing" }

<cti-ask> 表单 ​

模型现在可以让 Bridge 生成一个飞书 CardKit 2.0 表单，而不只是按钮卡片。协议示例：

xml
<cti-ask>{"question":"请选择发布策略","options":["灰度","全量"],"input":{"label":"补充说明","placeholder":"可留空"},"submitText":"提交","allowTextReply":true}</cti-ask>

渲染规则：

  • options 会渲染为一个 select_static。
  • input 会渲染为可为空的文本框。
  • 提交按钮使用 form_action_type: "submit"。
  • CardKit 2.0 提交值会出现在 raw event 的 action.form_value，key 是控件 name。因此表单控件使用固定 name：cti_choice 与 cti_input。

用户提交后，Bridge 会把回调转成同一会话里的下一条消息：

text
[用户回答了问题卡片]
问题：请选择发布策略
回答：选择：灰度
补充：可以先给 10% 用户

如果补充输入框为空，会显式回传 补充：（空），让模型知道用户是自由提交而不是消息丢失。

/doctor 时间参数 ​

/doctor 会读取 ~/.codex-to-im/logs/bridge.log，把日志片段和用户描述交给当前 Codex 会话分析。新增的第一个参数是时间范围：

text
/doctor
/doctor 10m 飞书卡片点击没反应
/doctor 2h bridge 重启后无法连接
/doctor 2026-06-02T01:00:00+08:00 云文档评论没有回复

时间参数支持：

  • 相对时长：10s、10m、2h、1d。
  • 可被 Date.parse() 解析的时间戳。
  • 不传时间时默认最近 30 分钟。

Bridge 会先在聊天里发一条 notice，说明读取了哪个时间窗口和多少行日志，然后把日志诊断 prompt 投递给当前会话。

交互卡片原始 JSON 注入 ​

“交互卡片原始 JSON 注入”指的是：当用户发送或引用一张飞书交互卡片时，不只把 SDK 抽出来的可见文字发给模型，还把卡片真实 JSON 包在一个专用块里发给模型：

xml
<interactive_card>
{ "schema": "2.0", "header": { ... }, "body": { ... } }
</interactive_card>

它解决的是 SDK 文本扁平化丢信息的问题。CardKit 2.0 事件里经常同时带有一个旧客户端 fallback elements 和真正的 user_dsl；如果只读 fallback，模型可能只看到“请升级至最新版本客户端”。原始 JSON 注入会优先把真实 user_dsl 或 schema 2.0 DSL 给模型。

这和 <cti-ask> 表单不是同一条链路：

  • <cti-ask> 是模型主动要求 Bridge 发一张受控问题卡片。
  • <interactive_card> 是用户侧已有卡片进入模型上下文，帮助模型理解那张卡的结构、按钮、字段和值。

当前实现入口在 Feishu adapter：

  • 用户直接发送 interactive 消息时，Bridge 会把该消息作为普通用户输入转给模型，并附上 <interactive_card>。
  • 用户引用一条消息作出回答时，Feishu receive event 的 message.parent_id 会被用于调用 im.message.get 读取被引用消息；如果被引用消息是 interactive，Bridge 会优先抽取 data.user_dsl，否则保留可解析的卡片 JSON，再放进专用的 <quoted_message> 引用块附加到模型 prompt。

用户引用一张交互卡片并回复“按这个窗口发”时，模型实际收到的 prompt 形态是：

text
按这个窗口发

<quoted_message platform="feishu" message_id="card-parent-1" message_type="interactive">
<interactive_card>
{
  "schema": "2.0",
  "body": {
    "elements": [
      {
        "tag": "markdown",
        "content": "发布窗口：今晚 22:00"
      }
    ]
  }
}
</interactive_card>
</quoted_message>

这里的第一段是用户本次发送的正文；空行后的 <quoted_message> 明确告诉模型“下面是被引用消息，不是用户新输入”。这个上下文只在普通模型输入阶段追加，不参与 /doctor、/t、/stop 等 Bridge 命令解析。
