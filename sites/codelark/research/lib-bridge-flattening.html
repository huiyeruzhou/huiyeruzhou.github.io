src 级自然聚合与 former bridge bucket 展平计划 ​

生成时间：2026-06-03

规划依据 ​

本计划使用历史 work/analysis/analyze-imports.mjs 作为采集工具，只取文件、行数、import 边、循环依赖等原始数据；不沿用旧脚本的 layer / cluster 聚合逻辑。

新的聚合逻辑来自当前代码的数据结构、文件名和实际 import 方向：

  • 产品持久数据实体：BridgeSession、ChannelChat、ChannelDefaultTarget、BridgeMessage、PermissionLink、OutboundRef、AutoTask、channel offset / dedup / audit。
  • Runtime 合同：LLMProvider、StreamChatParams、SSEEvent、TokenUsage、BridgeMirrorRecord、tool/task/context usage projection。
  • 应用状态机：interactive turn、mirror subscription、mirror turn、stream feedback、session health、adapter lifecycle、auto task loop。
  • 平台能力边界：Feishu、channel adapter capability、structured streaming card、callback / pin / preview / artifact delivery。

核心判断：former bridge bucket 不是唯一特殊的大模块。它之所以显眼，是因为 domain contracts、runtime contracts、channel contracts 和 bridge daemon application 都堆在这里。真正目标应该是先建立 src 级 owner，再把 former bridge bucket 收缩为兼容层并删除。

当前耦合事实 ​

历史脚本在当前树上采集到：

  • src/**/*.ts 文件数：253。
  • 本地 import / re-export 边数：1108。
  • former bridge bucket 相关文件仍是全仓最密集区域之一。
  • 多文件循环只有 host.ts <-> types.ts。

按当前数据结构重新分组后的事实：

关键耦合方向：

  • former bridge bucket -> configuration：25 条。
  • former bridge bucket -> codex-runtime：21 条。

  • operator-ui -> configuration：10 条。

具体 import 证据：

  • Codex / Claude provider 都直接 import src/domain/session.ts / src/runtime/contracts.ts 的 LLMProvider / StreamChatParams。
  • Codex session index parser import bridge types、mirror-source、context-usage、tool-call-details。
  • UI application 直接 import bridge session registry、display query、session runtime、transcript source。
  • store.ts 直接 import bridge contracts 和 session-runtime。
  • 已退役平台登录入口曾直接 import bridge adapter internals。

结论：只规划 former bridge bucket 展平会低估问题。应先把 src 下共享的数据结构和 runtime/channel 合同提升出来，再处理 bridge daemon 内部状态机。

全局 owner 逻辑 ​

1. domain ​

Owner：产品持久数据模型，不依赖 runtime、UI、adapter 或 store 实现。

应聚合：

  • BridgeSession、BridgeSession.runtime、session health fields。
  • ChannelChat、ChannelDefaultTarget、ChannelAddress。
  • BridgeMessage、InboundMessage、OutboundMessage、rich card DTO。
  • PermissionLink、permission callback storage DTO。
  • AutoTask。
  • TaskProgressInfo、ToolCallInfo、ContextUsageInfo、audit/dedup/outbound ref DTO。

当前来源：

  • src/domain/session.ts / src/runtime/contracts.ts
  • src/domain/index.ts
  • src/bridge/automation/auto-tasks.ts
  • src/shared/progress/context-usage.ts
  • src/shared/progress/tool-call-details.ts

判断标准：如果某个类型需要被 UI、store、bridge app、runtime provider、channel adapter 共同理解，它不属于 bridge app，应进入 domain。

2. runtime ​

Owner：模型运行时合同和具体模型 runtime 实现。

应聚合：

  • LLMProvider、StreamChatParams、SSEEvent、TokenUsage。
  • Runtime-neutral JSONL mirror source contract：MirrorJsonlSource、BridgeMirrorRecord。
  • Codex provider、Codex pty/tmux provider、Codex routing provider。
  • Codex local session index / JSONL parser / mirror source。
  • Claude pty/sdk provider、Claude session JSONL reader。
  • permission-gateway.ts、sse-utils.ts。

目标路径：

text
src/runtime/
  contracts.ts
  sse.ts
  permission-gateway.ts
  codex/
    provider.ts
    pty-provider.ts
    tmux-provider.ts
    routing-provider.ts
    session-index/
  claude/
    pty-provider.ts
    sdk-provider.ts
    session-jsonl.ts
    code-router.ts

判断标准：如果逻辑负责启动 Codex/Claude、解析 runtime 输出、恢复 runtime identity，或者定义 runtime 调用合同，它属于 runtime。它不应 import bridge app 的 host.ts。

3. channels ​

Owner：IM channel 合同、adapter runtime、平台实现、可靠 delivery。

应聚合：

  • Channel adapter contract 与 capability interfaces。
  • Adapter registry / sync plan / adapter lifecycle runtime。
  • Delivery layer、feedback delivery、stream feedback、outbound artifacts、callback data。
  • Feishu adapter、Feishu markdown/card/media/setup notice。
  • Weixin adapter、Weixin API/media/ids/session guard/types。

目标路径：

text
src/channels/
  contracts.ts
  adapter-runtime/
    registry.ts
    runtime.ts
    sync-plan.ts
    capabilities/
  delivery/
    deliver.ts
    feedback.ts
    stream-feedback.ts
    artifacts.ts
    callbacks.ts
  feishu/

判断标准：如果逻辑依赖平台 API、平台消息结构、平台卡片能力，或者负责把 domain message 送到 IM，它属于 channels。Bridge app 只通过 channels/contracts 和 channels/delivery 调用它。

4. bridge ​

Owner：Codex-to-IM daemon application workflow。它组合 domain、runtime、channels、storage、configuration，但不拥有这些层的数据结构。

应聚合：

  • Host lifecycle：start/stop/status/startup notice/inbound dispatch。
  • Interactive turn 状态机。
  • Mirror 状态机。
  • Slash command use-case 和 presentation。
  • Automation runtime。
  • Session health runtime。
  • Tmux remote-control command/runtime。
  • Doctor/diagnostics use-case。

目标路径：

text
src/bridge/
  host/
    manager.ts
    context.ts
    inbound-dispatch.ts
    startup-notice.ts
  turn/
  mirror/
  command/
  automation/
  health/
  diagnostics/
  tmux/

判断标准：如果对象生命周期是 daemon 运行时的一段 workflow，它属于 bridge。如果它是跨模块数据合同，就不属于 bridge。

5. storage ​

Owner：本地 JSON 存储、数据迁移、文件路径和 atomic write。

应聚合：

  • store.ts
  • storage-migrations.ts
  • CTI data path helper，但不拥有 config defaults。

目标是让 store 依赖 domain contracts，而不是 src/domain/session.ts / src/runtime/contracts.ts / types.ts / session-runtime.ts。

6. configuration ​

Owner：config.v2.json、env mapping、runtime/channel config schema、runtime option parsing。

应聚合：

  • config.ts
  • runtime-options.ts
  • channel instance config helpers。

目标是让 UI、bridge app、runtime provider 读取明确的 config query，而不是每层直接 import 大 config.ts 并复制解释规则。

7. operator-ui ​

Owner：本地 Web 工作台的 operator workflows。

当前 src/ui 已经部分聚合，但仍直接 import store/config/bridge internals。目标：

  • routes 只做 HTTP decode/encode。
  • application 层调用 storage/configuration/bridge/runtime 的 query/service facade。
  • 不直接 import bridge session registry internals。

最终可从 src/ui 改名为 src/operator-ui；也可保留路径但按上述 owner 收紧。

8. local-service ​

Owner：本地进程、开机自启、UI server process、包维护、Codex integration 安装。

当前 service-manager.ts 是 catch-all。目标拆分：

text
src/local-service/
  bridge-process.ts
  ui-process.ts
  autostart.ts
  package-maintenance.ts
  codex-integration.ts

CLI 和 UI service route 调用这些 workflow，不直接拼接平台脚本细节。

9. entrypoints ​

Owner：进程入口和依赖装配。

应聚合：

  • main.ts -> entrypoints/daemon.ts
  • cli.ts -> entrypoints/cli.ts

它们只做 wiring，不拥有业务规则。

former bridge bucket 的新定位 ​

在全局 owner 建立后，former bridge bucket 不应直接迁成 src/bridge。更准确的拆法是：

  • host.ts / types.ts 的跨模块合同先进入 domain、runtime/contracts、channels/contracts。
  • channel-adapter.ts 的平台无关 contract 进入 channels/contracts，具体 adapter 进入 channels/feishu。
  • mirror-source.ts 的 runtime-neutral source contract 进入 runtime/contracts 或 runtime/local-transcript，mirror 状态机进入 bridge/mirror。
  • session-runtime.ts 这种操作 BridgeSession.runtime 的 accessor 进入 domain/session-runtime.ts 或 bridge/session，取决于是否仍依赖 daemon 状态。当前更偏 domain helper。
  • bridge-manager.ts、interactive turn、mirror runtime、command、auto task、health 等 workflow 进入 src/bridge。

这比“把 former bridge bucket 整体搬到 src/bridge”更符合当前耦合事实。

迁移阶段 ​

阶段 1：建立全局 contracts，不再让其他模块 import former bridge bucket/host.ts ​

动作：

  • 新增 src/domain/*，承接 BridgeSession、ChannelChat、message、permission、auto task、progress/tool 等 DTO。
  • 新增 src/runtime/contracts.ts，承接 LLMProvider、StreamChatParams、SSEEvent、TokenUsage、MirrorJsonlSource、BridgeMirrorRecord。
  • 新增 src/channels/contracts.ts，承接 adapter base contract 与 channel capability interface。
  • 让 src/domain/session.ts / src/runtime/contracts.ts、types.ts、mirror-source.ts 暂时 re-export 新 contracts。
  • 迁移 Codex/Claude provider、store、UI application 的类型 import 到新 contracts。

验证：

  • npm run typecheck
  • 复跑依赖脚本，目标是 codex-runtime -> legacy-bridge、claude-runtime -> legacy-bridge、persistence -> legacy-bridge 明显下降。
  • 多文件循环 host.ts <-> types.ts 消失。

阶段 2：拆 storage / configuration owner ​

动作：

  • store.ts 迁到 src/storage/json-store.ts，只依赖 domain contracts。
  • storage-migrations.ts 迁到 src/storage/migrations.ts。
  • config.ts 拆出 configuration/index.ts、configuration/channels.ts、configuration/runtime-defaults.ts。
  • runtime-options.ts 迁到 configuration/runtime-options.ts 或 runtime/options.ts，按是否纯 Codex runtime 决定。

验证：

  • store/config/storage migration 测试。
  • UI config routes 测试。
  • npm run typecheck。

阶段 3：runtime provider 归位 ​

动作：

  • src/runtime/codex/* 迁到 src/runtime/codex/*，保留 src/runtime/codex/* facade。
  • src/runtime/claude/* 迁到 src/runtime/claude/*，保留 src/runtime/claude/* facade。
  • Codex / Claude 不再 import bridge app；只 import domain、runtime/contracts、configuration、shared。
  • Codex session-index 中的 tool/context projection 若跨 runtime 可复用，放入 domain/progress.ts 或 runtime/transcript-projection.ts。

验证：

  • Codex provider / routing / session-index / mirror 测试。
  • Claude provider / session-jsonl 测试。
  • npm test 中 runtime 相关套件。

阶段 4：channels 归位 ​

动作：

  • src/channels/feishu/adapter.ts 拆迁到 src/channels/feishu。
  • 已退役平台 adapter、登录和账号存储入口直接删除，不再迁入新 owner。
  • delivery-layer.ts、feedback-delivery.ts、stream-feedback-controller.ts 迁到 channels/delivery。
  • Adapter runtime / sync plan 迁到 channels/adapter-runtime。

验证：

  • Feishu adapter / markdown / card e2e tests。
  • Weixin adapter / API / media / login / store tests。
  • delivery pipeline / stream feedback tests。

阶段 5：bridge daemon workflows 归位 ​

动作：

  • bridge-manager.ts 收缩为 src/bridge/host/manager.ts。
  • interactive-turn/* + turns/* 进入 src/bridge/turn。
  • mirror-* 的状态机进入 src/bridge/mirror，但 mirror source contract 留在 runtime。
  • command/* 进入 src/bridge/command，只调用 owner facade。
  • auto-tasks.ts 中数据 contract 留在 domain/automation.ts，runtime loop 进入 src/bridge/automation。
  • session-health-* 进入 src/bridge/health。

验证：

  • bridge manager / command / interactive turn / mirror / health / automation 相关测试。
  • 完整 npm test。

阶段 6：operator UI 和 local service 收口 ​

动作：

  • UI application 不再直接 import bridge internals；改用 domain DTO + storage/config/runtime/channel query facade。
  • service-manager.ts 拆到 src/local-service/*。
  • main.ts / cli.ts 迁到 src/entrypoints/*，只做 wiring。

验证：

  • UI route/application tests。
  • service-manager tests。
  • build / typecheck / full test。

阶段 7：删除兼容层 ​

动作：

  • 删除 former bridge bucket re-export facade。
  • 删除 src/codex / src/claude / src/ui 等临时 facade，或明确保留为 public alias。
  • 扫描所有旧路径。

验证：

  • 扫描源码、文档、脚本、schema 中的旧 bridge bucket 和旧 runtime facade import 字符串。
  • npm run typecheck
  • npm run build
  • npm test

推荐第一刀 ​

第一刀不建议再以 bridge/session/contracts 为中心。应做：

  1. domain contracts：session、channel、message、permission、automation、progress/tool。
  2. runtime/contracts：LLM provider、stream params、SSE、mirror source/record。
  3. channels/contracts：adapter base contract 和 capability interfaces。
  4. src/domain/session.ts / src/runtime/contracts.ts、types.ts、mirror-source.ts 改为兼容 re-export。
  5. 迁移 Codex/Claude provider、store、UI application 的类型 import。

理由：这几类模块现在都通过 former bridge bucket 互相耦合。先抽全局 contracts，才能让后续 former bridge bucket 展平不变成“把 bridge 换个名字继续当公共桶”。

不做什么 ​

  • 不沿用历史脚本的聚合分类作为目标边界。
  • 不把 bridge/core、cluster-01、cluster-02 这类审计标签变成目录名。
  • 不把 former bridge bucket 整体搬到 src/bridge 后继续承载所有共享类型。
  • 不让 runtime provider import bridge daemon application。
  • 不让 storage import bridge app contracts。
  • 不让 operator UI 直接读取 bridge session registry internals。
  • 不新增大而空的 composition / factory / deps 层。
  • 不把 bridge-manager.ts 的复杂度搬到另一个大 manager。
