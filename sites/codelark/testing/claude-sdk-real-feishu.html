Claude SDK 真实飞书验证计划 ​

本文只记录 Claude SDK/无头 Claude provider 的专项背景和验收要求。真实飞书 E2E 的通用运行方式、场景组织、gate 和排障流程见 真实飞书 E2E。

背景 ​

2026-06-02 的 Claude runtime 排障暴露了一个核心问题：Claude Code 已经写出 JSONL 响应，但 BridgeSession.runtime.claude.sessionId 保存的是 bridge 自己的 session id，而不是真实 Claude Code session id。结果 mirror reconciler 认为 Claude thread 不存在，用户在 IM 里看到的是“Claude 不回复”。

典型日志形态：

text
Clearing dangling Claude thread f63c88b5-0cca-4c5d-ad91-58d1e69f02c9:
Claude thread no longer exists locally

真实故障不是模型没有输出，而是 bridge 没有把 IM session 绑定到真实 Claude JSONL session。

现有 pty 路径的脆弱点 ​

当前 pty provider 在注入 prompt 后再猜测 Claude 身份：

  1. 启动或复用 Claude Code TUI pty。
  2. 把 prompt 写入 TUI。
  3. 等待 CTI_CLAUDE_PTY_JSONL_DISCOVERY_TIMEOUT_MS，默认 2500ms。
  4. 扫描 ~/.claude/projects/<cwd>/*.jsonl，查找注入 prompt 后更新的文件。
  5. 找到后发送 status { session_id, cwd, transcript_path }。
  6. interactive runner 写入 runtime.claude.sessionId，再触发 mirror reconcile。

如果短窗口内没有扫到文件，status 事件不会发出，runner 不会激活 Claude mirror 输出，session 可能长期绑定到无效占位 id。

SDK 与无头 CLI 能力 ​

Claude CLI / ccr code 暴露了这些与会话身份相关的能力：

  • --session-id <uuid>：使用指定 session id。
  • --resume [value]：按 session id 恢复对话。
  • --continue：继续当前目录最近一次对话。
  • --fork-session：恢复时 fork 出新 session。
  • -p, --print：非交互输出模式。
  • --output-format stream-json：输出结构化流。
  • --input-format stream-json：接收结构化流输入。

Claude Agent SDK 也提供更适合 bridge 的事件路径：

  • TypeScript 包为 @anthropic-ai/claude-agent-sdk。
  • query() 支持 string | AsyncIterable<SDKUserMessage>。
  • streaming input 支持排队消息、中断、权限请求、图片和持久 session。
  • 可用 slash command 会在 system/init 消息的 slash_commands 中暴露。
  • SDK 暴露 listSessions()、getSessionMessages() 等 session 工具。

Provider 设计方向 ​

bridge 不应把 Claude 身份当成事后猜测结果。Claude runtime 身份应在 turn 初始化前后明确获得。

目标行为：

  • 新增 Claude SDK 或 headless provider。
  • IM Claude runtime turn 可按 provider 配置选择 pty 或 sdk。
  • pty 保留为 attach/debug/CCR 场景；SDK provider 作为原生事件路径。
  • 首次 turn 没有 Claude session id 时，生成 UUID，或立即读取 SDK init/result 中的 session id。
  • 把真实 id 写入 runtime.claude.sessionId。
  • 新 session 使用 --session-id <uuid> 或 SDK session 选项；已有 session 使用 resume / --resume <session-id>。
  • 把 SDK 消息转换为 bridge 现有 SSE event vocabulary。
  • SDK 报告 session id 后立刻发送 status，身份明确后触发 mirror reconcile。
  • 从 SDK stream 中发出 text/tool/task/context/result 事件。

当前实现状态：

  • runtime.claude.provider、CTI_CLAUDE_PROVIDER、bridge_claude_provider 和会话级 runtime.claude.provider 支持 pty|sdk。
  • /provider 和 /p 在 Claude runtime 下切换 pty|sdk；Codex runtime 下仍切换 sdk|pty|tmux。
  • sdk provider 基于 @anthropic-ai/claude-agent-sdk query()，把 init/result/assistant 文本转成 bridge 现有 SSE vocabulary。
  • 默认仍是 pty，避免改变当前 CCR-backed Claude Code 的真实飞书路径。

Slash command 边界：

  • /t、/his、/runtime、/new、/auto、/file、/shell 等仍由 Codex-to-IM 处理。
  • 只有 bridge command dispatch 拒绝后，Claude slash command 才传给 Claude。
  • provider 应把 SDK slash_commands 列表暴露给诊断。
  • SDK 模式不支持的命令应给出可见诊断，不能静默退回 pty 行为。

真实飞书验收要求 ​

Claude SDK 真实飞书场景必须满足通用 真实飞书 E2E gate，并额外证明：

  • claude-sdk 使用 SDK 原生事件和 im: direct stream key，不要求 claude_jsonl_found。
  • claude-pty 才要求 Claude JSONL/mirror 证据，JSONL 查找必须兼容 Claude Code 对 cwd 的编码规则。
  • SDK final prompt 的 bot 回复本身必须包含本轮唯一 marker，不能把用户原始消息里的 marker 当成通过证据。
  • CCR-backed claude-sdk 测试必须通过 ccr activate/status/start 准备 SDK subprocess env。
  • 使用 --fake-ccr 时，必须同时证明 fake backend 收到请求，并且最终 bot reply_to 包含 fake marker。
  • fake 代理只提供可控模型输出，不能替代 provider 输出路径、reply_to 和清理 gate。

未决问题 ​

  • pathToClaudeCodeExecutable 是否可以直接指向 ccr。
  • SDK 能否加载 CCR 的同一套配置，而不是 shell 到 ccr code。
  • 对 CCR 兼容而言，headless CLI 是否比 TypeScript SDK 更稳。
  • SDK 权限请求如何映射到现有飞书权限卡片。
