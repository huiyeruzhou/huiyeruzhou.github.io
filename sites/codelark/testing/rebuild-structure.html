测试代码结构重建计划 ​

本文记录测试目录和测试职责。源码已按 owner 展开后，测试也按 owner、验证强度和公共工具边界分层，避免继续把所有 suite 平铺在 src/__tests__。

重建目标 ​

测试结构应同时回答三个问题：

  1. 这个测试保护哪个源码 owner。
  2. 这个测试使用哪一级验证强度。
  3. 这个测试是否可以作为用户故事或发布 gate 的证据。

新结构不追求把每个源码文件逐一镜像成一个测试文件。功能点非常单一、没有跨模块状态的逻辑，只保留单测；一个 owner 内有多个协作点时，配一个本地打包 workflow/e2e；跨一大组 owner、依赖 provider 启动或 CLI 协议时，配真实可执行文件加 proxy 的本地 e2e；跨多个模块、全局状态切换且用户可见的完整故事，交给真实飞书 E2E。

src/__tests__/unit 与源码 owner 对齐。workflow 用 fake adapter/provider/store 打包一个 owner 内或相邻 owner 的协作链路。e2e/mock-app 是本地可重复的应用级 e2e，可以使用 mock provider 和 mock channel。e2e/local-process 必须启动真实可执行文件或真实进程边界，并通过本地 proxy/fake backend 控制模型结果。harness/real-feishu 测真实飞书 harness 自身，真实飞书长流程仍由 scripts/real-feishu-e2e.ts 执行。

fixtures/ 只放可被多个测试层级复用的静态输入数据，例如 config JSON、Codex/Claude JSONL、飞书 payload 和 transcript。只被单个 suite 使用的数据应放在该 suite 旁边，避免全局 fixtures 变成第二个无 owner 的杂物目录。动态生成的 store、home、proxy 和 adapter 不属于 fixtures，应放在 setup/ 或 helpers/。

目前尚未引入全局 fixtures/ 目录；现有静态输入仍就近保留在对应 suite 中。只有当同一份静态输入跨多个层级复用时才新增 fixtures/。

强度分层 ​

层级目录依赖应证明的事不应承担的事
单测unit/<owner>/纯函数、小对象、临时文件单一解析、转换、状态 reducer、presentation、schema、路径规则正确不证明用户故事，不 mock 一整条聊天流程
本地 workflowworkflow/<owner-or-slice>/fake adapter/provider/store一个 owner 或相邻 owner 组合后能完成一组稳定动作不启动真实 Codex/Claude，不依赖飞书
本地 mock e2ee2e/mock-app/<slice>/bridge host + fake channel + fake providerdaemon 级编排、绑定、状态持久化、交付动作能闭环不证明 provider 可执行文件或飞书客户端契约
本地真实进程 e2ee2e/local-process/<runtime>/真实 codex/claude/tmux 可执行文件 + 本地 proxy/fake backendprovider 启动、CLI 参数、JSONL/mirror、清理和 provider 输出路径可信不证明飞书事件投递、reply_to 或真实卡片渲染
真实飞书 E2Escripts/real-feishu-e2e.ts 场景真实 lark-cli、真实群、真实 bridge、可控 provider 后端完整用户故事、跨模块全局状态、真实飞书消息/卡片/文件/清理 gate不逐条镜像单测和本地 workflow

判断规则：

  • 一个函数或 formatter 的输入输出足够描述风险时，只写单测。
  • 一个命令组、session registry、delivery pipeline、mirror controller 等包含多个内部组件时，保留同组 workflow。
  • 一个 runtime/provider 功能涉及真实 pty、tmux、CLI bootstrap、JSONL 文件或 proxy 协议时，必须有 e2e/local-process。
  • 用户从飞书发消息开始，经历 runtime/provider 切换、跨群绑定、历史、表单、附件、权限或清理时，只能由真实飞书 E2E 作为最终证据。

bridge/command 拆分标注 ​

bridge/command 需要源码重构和测试重构同步推进。当前 session-thread.ts、runtime-settings.ts、tmux.ts、dispatch.ts 等文件同时承担命令识别、参数解析、权限/状态读取、业务动作、presentation、delivery 后处理和真实副作用编排。重建时先按现有自然功能块拆出最小必要边界，再让测试目录跟随这些边界；不要为了目录整齐把每条命令都拆成独立目录。

dispatch.ts 是命令入口和交付编排，result.ts 只定义命令处理器返回给 dispatch 的结构化结果。aliases.ts、callbacks.ts、clear-confirmations.ts、errors.ts 这类小而稳定的文件可以留在根部，除非后续确实出现多个文件共享同一职责。commands/ 也不按每个 slash command 建目录，而是按修改半径和上下文聚合：session-thread、runtime settings、diagnostics、auto、tmux/pty/shell、control、hot update、global settings、require-at。命令组如果仍能用一个文件表达清楚，就保持单文件；只有当该组已经拆出 parser、use-case、side-effect adapter 等多个低层文件时，才升级成目录。

拆分顺序应从最大、最混杂的文件开始：先把 session-thread.ts 中的 list/card/history presentation 和 session use-case 分开；再把 runtime-settings.ts 中纯 formatter、bootstrap helper 和设置命令处理器分开；然后把 tmux.ts/pty.ts/shell.ts 的 parser、screen monitor、runner 和 command handler 边界收窄；最后收缩 dispatch.ts。dispatch 不一定要一次性变成注册机制，但应逐步只保留危险输入校验、命令路由、context 构造、CommandResult delivery 和审计。这样单测可以覆盖 parser/presentation/use-case，workflow 覆盖命令组状态闭环，mock e2e 覆盖 bridge 集成，真实飞书只覆盖完整用户故事。

当前源码重构采用模块边界优先的完成标准：bridge/host 是生产代码里唯一直接调用 bridge/command 的上层 owner；channels、bridge/turn、runtime、storage、operator-ui 不直接依赖 command；command 也不反向 import bridge/host/*。跨 turn 和 command 共用的 agent question callback 协议位于 src/bridge/callbacks/agent-question.ts，聊天绑定 router 位于 src/bridge/session/channel-router.ts，startup notice target 位于 src/bridge/startup-notice-target.ts。后续如果继续拆 dispatch.ts，应在这个单向依赖边界内演进，而不是为了目录整齐继续细拆每条 slash command。

删除和降级规则 ​

重建时应主动删除冗余测试，而不是只移动文件。

可删除：

  • 同一输入输出已经由更低层单测完整覆盖，且高层测试只重复断言同一个字符串。
  • 旧目录、旧平台或已删除 facade 的兼容性测试。
  • 只断言 mock 被调用一次、但没有检查状态变化或用户可见结果的测试。
  • 与真实飞书 harness metadata 不一致的旧场景快照。

应降级或拆分：

  • 一个大型 workflow 里混入多个纯 formatter/parser 断言，应把纯逻辑拆到 unit。
  • 一个 mock-only 测试声称证明完整用户故事，应改名为 workflow 或 mock-app e2e，并在覆盖审计中标明不能替代真实飞书。
  • 一个真实进程 e2e 只用了 fake executable，没有启动真实 provider 边界，应降级到 workflow/runtime。

必须保留或补强：

  • 覆盖隔离 home、真实进程清理、tmux 清理、created chat 清理和 report gate 的测试。
  • 覆盖 provider 输出路径的测试：SDK 应证明直接 im:，pty/tmux 应证明 mirror:，并检查最终文本不重复。
  • 覆盖飞书文件、交互卡片、Markdown 原始内容和 reply_to 的真实平台证据。

命名和运行入口 ​

命名建议：

  • 单测：*.unit.test.ts，也可以在迁移期保留 *.test.ts。
  • 本地 workflow：*.workflow.test.ts。
  • 本地 mock app e2e：*.mock-e2e.test.ts。
  • 本地真实进程 e2e：*.local-e2e.test.ts。
  • 真实飞书 harness 自测：*.harness.test.ts。

未来 scripts/run-tests.js 应从固定读取 src/__tests__/*.test.ts 改为递归发现，并支持按层级过滤：

bash
npm test
npm test -- --unit
npm test -- --workflow
npm test -- --mock-e2e
npm test -- --local-e2e

真实飞书继续使用：

bash
npm run real:feishu:e2e -- --list-scenarios
CTI_REAL_FEISHU_E2E=1 npm run real:feishu:e2e -- --scenario basic-dialogue-suite

迁移顺序 ​

  1. 先引入递归 test discovery 和 setup/，保持现有测试全部可运行。
  2. 迁移公共工具：拆出 isolated home、recording adapter、fake provider、runtime proxy 和 assertions。
  3. 迁移纯逻辑单测：shared、domain、configuration、storage、小型 bridge controllers。
  4. 迁移 workflow：command、session、turn、mirror、delivery、UI route。
  5. 迁移本地 e2e：mock app bridge、real Codex/Claude/tmux provider。
  6. 清理冗余测试，并更新 coverage-audit.md 中的本地覆盖和真实飞书证据。

每一步都必须保持 npm test、npm run typecheck 和 npm run docs:build 可作为阶段验证。涉及真实飞书矩阵或场景 metadata 时，还要检查 npm run real:feishu:e2e -- --list-scenarios 输出与文档一致。
